## Architecture: what's wired and how

### Layer map

```
WhatsApp <─ws─> daemon (Baileys)
                  │ commands (Redis Streams)        │ events (Redis Pub/Sub)
                  ▼                                 │
       api (Fastify) <──────────────────────────────┘
       │ REST + SSE
       ▼
       web (Vite + React)
```

### Dev workflow

```fish
# 1. Postgres + Redis (containers)
docker compose -f docker-compose.local.yml up -d

# 2. Migrations
pnpm --filter @yank/db drizzle:migrate

# 3. Daemon + api + web (single terminal, colored prefixes)
pnpm dev
```

Tail logs (optional, lets a future agent grep without re-pasting):
```fish
pnpm dev 2>&1 | tee /tmp/yank-dev.log
```

`.env` at repo root needs (template in `.env.example`):
- `DATABASE_URL`, `REDIS_URL`
- `YANK_USER_ID` (UUIDv7, hard-coded single user in v1)
- `YANK_BAILEYS_AUTH_DIR=/home/<user>/.yank-baileys` (NOT `$HOME/...` — node's `--env-file` doesn't expand vars)
- `YANK_MEDIA_DIR` defaults to `$(node -p "require('os').tmpdir()")/yank-media`

### Linking flow

1. User navigates to `http://localhost:5173/setup`
2. SetupView queries `/api/setup/status`; if `unlinked`, fires `POST /api/setup/link { method: 'qr' }`
3. API publishes `pair` command to Redis
4. Daemon's connector starts Baileys, emits `connection.update` with `qr` data
5. Daemon publishes `qr` event on SSE; SetupView renders QR via `qrcode.react`
6. User scans on phone
7. Baileys fires `connection.update` with `connection: 'open'`; daemon upserts `whatsapp_sessions` row, publishes `connected` event
8. SetupView transitions through `connected → syncing → done` as events arrive
9. "Open Yank →" → `/` redirects to last-active chat in current workspace

## Gotchas — read these before debugging anything WA-related

1. **WhatsApp anti-abuse cooldowns.** Multiple daemon restarts in a short window will throttle this account. Symptoms in the log:
   - `"Timeout in AwaitingInitialSync, forcing state to Online and flushing buffer"`
   - Media reupload requests never get a response → our 30s timeout fires
   - Group metadata calls succeed but `chats.upsert`/`contacts.upsert` events never fire
   Fix: wait. Hours, sometimes a day. Restarting the daemon makes it worse, not better.

2. **Pair-time-only data.** WhatsApp ships your address book (contact display names) and your chat list ONCE at link time. Subsequent reconnects don't re-send. If WA was being slow at link time and the initial data dump didn't include names, you're stuck with sparse `contacts.display_name` until either:
   - Live messages arrive (each carries `pushName` we write to contacts)
   - You re-pair (wipes creds + DB needed; expect another cooldown)
   - A manual contact-rename UI is added (recommended M4 task; small + high impact)

3. **Media expiry is real.** WhatsApp's CDN keeps media bytes for weeks-to-months, not forever. Old history-sync messages will often return error code 2 from `updateMediaMessage` ("Failed to re-upload media (2)"). The current code correctly marks these as `failureReason: 'expired'` and stops retrying. Live media (received from now on) works fine.

4. **Baileys 6.7.21 retry quirks:**
   - `downloadMediaMessage` only auto-retries on HTTP 404/410, not 403 (the most common stale-URL response). Our `BaileysConnector.downloadMedia` wraps `updateMediaMessage` manually for 403s.
   - It requires a pino-compatible logger in the ctx — passing `undefined` crashes on the retry path. We pass the daemon's real pino logger now.

5. **`messages.text` is nulled on REVOKE.** When a message is deleted-for-everyone, the daemon does `UPDATE messages SET deleted_at = NOW(), text = NULL WHERE wa_message_id = ?`. The web renders the tombstone based on `deletedAt != null`. Don't rely on `text` to know the original content of deleted messages — it's gone.

6. **`chats.subject` is null for groups during the first hour or two of a new pair.** WhatsApp's chat-metadata events arrive piecemeal. Our daemon backfills via `sock.groupMetadata(jid)` for every group on connection.open (rate-limited at WA's edge — patience).

7. **`@lid` addresses.** WhatsApp 2024+ introduced anonymized "linked identifier" addresses for users who haven't shared their phone number in a group. Our DB stores these as-is. There's no Baileys API to resolve `@lid` → display name; their `pushName` may also be sparse. Treat them as opaque for now.

## Known limitations / suggested M4 starting points

Per the original M3 plan's "What's NOT in M3 (deferred)" list, plus things we discovered along the way:

### High-impact, small effort
- **Manual contact-rename UI** — click DM title in ChatTopbar to inline-edit; writes `contacts.display_name`. The only reliable way to fix sparse contact names. Small UI change + endpoint already exists in spirit (we have api routes for chat-level edits).
- **Circuit breaker for media downloads** — when many consecutive `updateMediaMessage` calls timeout, stop accepting new download commands for a cooldown window. Otherwise the IntersectionObserver can re-flood WA when the user scrolls past visible queued media.
- **WhatsApp degradation banner** — when daemon publishes a `disconnected` event, show a top-of-shell banner. Event already flows; just no UI.

### Per the spec / M1 deferral list
- **Triage card grid** (M4) — the "swipe through each new chat, assign with 1/2/3" UX from the mockup. Today's experience: every chat lands in Triage, user has to click the workspace pill in the topbar per chat. The card grid is the real UX. Route stub exists at `routes/triage.tsx`.
- **Search results + filter chips** (M5) — route stub exists. FTS GIN indexes are already in the schema.
- **Saved messages** (M5) — route stub exists. `stars` table in schema. `useStar` mutation exists.
- **@mention autocomplete in composer** (M4) — needs `useChatMembers` wired into a popover.
- **Edit-last shortcut (`↑` in empty composer)** (M4) — needs edit-message API + protocolMessage EDIT support.
- **Hover shortcuts** (`R` reply-in-thread, `S` star) (M4).
- **More keyboard shortcuts**: `Cmd-T` quick-switcher (chats only), `Cmd-F` search-current-chat, `Cmd-Shift-A` mark-current-read (M4).
- **In-page `Notification.show()` for inbound messages while tab is unfocused** (M6, alongside Web Push).

### Per the design spec §12, deferred from v1 entirely
- Multi-user auth (single hard-coded user in v1)
- Polls voting / Status / Calls UI / Newsletters
- Backfill from `.txt` chat exports
- Light-theme polish (dark theme is the production target)

## Open questions to surface to the human at M4 kickoff

- **Should `feat/m3-frontend` be merged before M4 starts?** It's 78 commits and adds a lot of scope beyond the planned M3. A merge cleans the slate but loses the granular commit boundary if anything regresses.
- **Manual contact rename: separate M4 feature or merge into Triage flow?** Triage is "decide where this chat lives"; renaming a DM at triage time would make the rest of the app instantly more usable.
