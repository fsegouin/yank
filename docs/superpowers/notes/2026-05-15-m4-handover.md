## Gotchas discovered during M4

1. **Root vs web vitest configs.** Root vitest at the repo level uses node env and only matches `*.test.ts`; the web package config uses jsdom and matches `*.test.{ts,tsx}`. Tests using `@testing-library/react` MUST end in `.tsx` or they will fail at the root suite (no `document`). Bit `useMentionAutocomplete.test.ts` (renamed `.tsx`) and `eventStream.contact.test.ts` (renamed `.tsx`).

2. **`@yank/shared` barrel exports.** Phase 0 added schemas to `dto.ts` but didn't re-export from `index.ts`. Tasks 1.3 and 2.1 had to add the re-exports retroactively. New schemas should always be added to BOTH `src/<file>.ts` AND `src/index.ts`.

3. **Phase 0 `ContactUpdateEvent.contactId` schema bug.** Was `z.string().uuid()` but the contacts table uses composite PK `(userId, jid)`. Task 2.1 fixed this to `z.string().min(1)`; the event now carries a JID.

4. **`chats.id` has no DB default.** Every test that inserts a chat row must supply `id: newId()` explicitly. Hit in Tasks 1.3 and 2.1.

5. **TanStack Query v5 `onMutate` is async-aware.** Tests using `renderHook` + optimistic-patch assertions must use `await act(async () => {...})` even when the `onMutate` callback is synchronous; pure-sync `act(() => {...})` doesn't flush the microtask queue.

6. **Pre-existing send-path `replyToId` bug surfaced.** The web sends `SendMessageBody { text, replyToId? }` but the api was previously reading `req.body.quotedWaId` (different name, different semantics). Task 3b.3 cleaned this up — the api now validates with `SendMessageBodySchema.safeParse` — but the `replyToId → waMessageId` lookup is still NOT wired. Quoting via the web is not functional in v1; M5 should add the lookup.

7. **React 19 + Zustand object selectors.** Selectors that return `{ a, b }` objects on every call cause infinite loops in React 19 (`getSnapshot should be cached`). Use individual primitive selectors instead. Bit both `MediaPausedChip` and `MediaImage`; resolved by switching from a `useMediaBreakerState()` helper to `useMediaBreakerStore((s) => s.state)` + `useMediaBreakerStore((s) => s.retryAt)`.

8. **Daemon circuit-breaker resets on process restart.** State is in-process; on daemon restart, the breaker is `closed` regardless of pre-restart state. Acceptable for v1 — the breaker re-trips quickly if WA is still throttling. The Redis hash `breaker:user:<userId>` persists last-known state for fresh-tab reconciliation but does not restore the in-memory window.

9. **WhatsApp 15-minute edit cliff.** Baileys rejects edits on messages older than ~15 min with `'too-old'` reason. The UI does not gate this in advance — the user sees a per-row retry affordance. Post-v1: add a tooltip on the Edit menu item.

10. **`bypassBreaker` not fully wired end-to-end.** The web passes `?bypass=1`; the `DownloadMediaCommand` schema already includes `bypassBreaker?: boolean` (added in Task 4.2). However the api route in `media.ts` does not yet forward `bypassBreaker: true` into the `XADD` command payload when `?bypass=1` is present. Small follow-up for M5.

## Suggested M5 starting points

Per spec §2 deferred list and what we learned in M4:

1. **Search results** — `/search` route stub exists; FTS GIN indexes are already in the schema. M5 can add `GET /api/search?q=<text>` using PostgreSQL full-text search (`to_tsvector` + `to_tsquery`). The M4 `<ChatFilterBar>` (`Cmd-F`) is window-bound — cross-chat FTS is M5.

2. **Saved messages view** — `/stars` route stub; `stars` table + `useStar` mutation already exist. M5 renders a message list similar to `ChatView` reading from `stars`.

3. **Wire `replyToId → waMessageId` lookup** in the send path so quoting works end-to-end (see gotcha #6 above).

4. **`bypassBreaker` wiring** — wire `bypassBreaker: true` from `GET /api/media/:id?bypass=1` into the `XADD` command payload (see gotcha #10 above; small but real gap).

5. **Circuit-breaker env-var tuning** — expose `YANK_BREAKER_THRESHOLD`, `YANK_BREAKER_WINDOW_MS`, `YANK_BREAKER_BASE_COOLDOWN_MS` so production tuning doesn't require code changes.

6. **Per-chat re-assignment menu** in `ChatTopbar` (workspace dropdown from any chat view; currently only available in TriageCard).

7. **`ChatTopbar` click-to-rename** — currently only `TriageCard` has `<InlineRename>`; the same primitive can be reused in `ChatTopbar` for DMs.

8. **Edit history preservation** — currently overwrites `text` in place. Post-v1 wants the original retained (new `message_edits` table or `jsonb` history column).

9. **`/hidden` recovery view** — currently `hidden` chats are invisible everywhere. M5 can add a recovery view at `/hidden` (simple list with "Restore" action).

10. **Group avatars** — `Avatar` uses `avatarGradient`. Real WA profile-picture data from Baileys can be persisted to `contacts.avatar_url` and rendered in M5.

- Should the live-api Playwright project run in CI, or stay as a manual smoke? (Currently requires a running daemon + api + Postgres + Redis to pass.)

- Should group avatars fetch from WA CDN or be stripped entirely for privacy? CDN fetch needs a media-worker-style proxy.

- `paletteMode` and `chatFilter.open` are both UI-store booleans — should there be a single "ephemeral overlays" slice managing all overlay state?
